# NWK 설계 방향 — 공공데이터 중심 서비스 근거

> 2026 관광데이터 활용 공모전 제안서 집필 시 참조 문서.
> 본 문서의 톤은 공식 평서체이며, 앱 UI의 `~어요` 톤과 의도적으로 분리한다.

---

## 1. 설계 전제 — 데이터 우선, 지역은 결과

NWK의 핵심 기획 분기점은 **"외국인 관광객에게 공공데이터 기반의 가격 참조 도구를 제공할 수 있는가"**였다. 전국 단위의 균질한 가격 공공데이터는 존재하지 않았고, 지자체별 개인서비스요금 공개 여부가 극히 분산되어 있었다.

조사 결과, **경기도 부천시(data.go.kr 3079356, UDDI `59f52e8f-2d39-4c45-909e-97d413996b66`)** 가 아래 세 조건을 모두 만족하는 드문 사례였다.

- **Open API 형태로 공개** (스크레이핑이 아닌 정식 호출 가능)
- **갱신이 지속적으로 유지**되는 메타데이터
- **항목 분류 명확** (품목·규격·지역·조사 시점)

따라서 **부천시 데이터셋의 존재가 지역 선정을 결정**했다. 부천시 자체의 관광 매력 때문이 아니라, 공공데이터 기반 정상가 도구 구축이 가능한 실증 베드가 부천시였기 때문에 선정되었다는 순서가 중요하다.

### 2.4 TourAPI 네 개 언어 서비스 병렬

- `KorService2` / `EngService2` / `JpnService2` / `ChsService2` 네 개 서비스를 동일 엔드포인트로 순회
- `contentId`를 merge 키로 하여 하나의 POI 레코드에 네 개 언어 번역이 결합됨
- **5개 엔드포인트 × 4개 언어 서비스 = 20개 조합 실증**

### 2.5 수동 번역 레지스트리 (보완 장치)

- TourAPI의 외국어 서비스에 번역이 누락되는 POI가 실제로 다수 존재함을 확인 (부천 일대 중소형 POI)
- `src/data/poi-translations.json`에 **40개 POI × 3개 언어 (영·일·중) 수동 번역** 시드
- fallback chain: `requested language → manual registry → TourAPI(lang) → Korean`

## 3. `isLive` 플래그의 정확한 의미

UI에서 표시되는 **LIVE 뱃지**는 흔히 오해되는 "실시간 API 호출"을 뜻하지 않는다. 코드에서 `isLive = true`는 아래 한 가지 경우만 가리킨다.

> **해당 품목이 부천시 개인서비스요금 공공데이터셋에 존재해, 내부 참조 카탈로그가 아닌 공공데이터 수치로 비교가 이뤄졌음.**

즉 `LIVE = 공공데이터 직결`이며, 실시간 호출 여부와는 직교 축이다. 가격 데이터는 **빌드 타임 스냅샷**으로 관리하며, 이 설계는 아래 세 가지 이유로 의도되었다.

1. **해외 IP 차단 완화** — TourAPI 일부 엔드포인트와 data.go.kr 제공 API는 해외 IP에서 403을 반환하는 사례가 관찰됨. 외국인 관광객이 실 사용자인 본 서비스에서 런타임 호출은 리스크가 크다.
2. **지연 최소화** — 가격 비교는 페이지 로드 시점에 0ms에 근접해야 UX가 성립한다.
3. **비용 예측 가능성** — 런타임 호출은 Cloud Functions 과금과 data.go.kr 쿼터에 직접 노출됨.

TourAPI 검색(`searchKeyword2`)은 임의 키워드 대응이 필요해 예외적으로 런타임으로 유지한다. 단 결과에 `source: 'live' | 'mock'` 필드를 붙여 실제 원본 호출 여부를 시연에서 증명 가능하게 했다.

## 4. 내부 참조 카탈로그의 역할

`src/data/price-catalog.ts`는 **부천시 공공데이터셋이 커버하지 못하는 품목의 보조 참고값**이다. 대표적 예:

- **국제 관광객 전용 품목**: 빙수·버블티·붕어빵 (부천 조사본 외)
- **수도권·전국 단위 교통**: KTX·공항철도·렌터카·전동킥보드 (지역 가격 개념 없음)
- **숙박 등급별 구분**: 4성급 호텔·게스트하우스·펜션 (부천 조사본은 단일 "숙박" 항목만)
- **체험·기타**: 테마파크·케이블카·PC방·짐보관·주차

운영 원칙:

- 공공데이터에 존재하는 품목은 **자동으로 공공 값이 우선** (`getLiveOverride()`)
- 내부 참고값은 **이력 추적 가능하게 명시** (출처 코멘트 함께 유지)
- UI에서 두 경우를 **`isLive` 뱃지로 시각 구분**

## 5. 코스 큐레이션 — 70% 부천 원칙

각 코스는 **전체 스톱의 70% 이상이 부천시 내에 위치**해야 한다 (`validateBucheonShare()`).

- 인근 최대 30%는 서울 지하철 1·7호선·공항철도 환승권 내 시군구로 제한 (광명·김포·부평·강서·구로)
- 목적: "부천 전용"이 아닌 **"부천 거점 + 수도권 접근성 동시 홍보"**
- 31개 전 코스가 이 원칙을 통과하며, validator는 런타임에도 항상 실행됨

## 7. 공모전 심사 기준과의 매핑

### 1차 심사 100점

- **서비스 기획력 (30점)** — 기획 배경이 "공공데이터 존재 → 지역 선정"이라는 **데이터 우선 설계**임을 제안서 p1~p2에서 명시
- **서비스 완성도 (30점)** — 31개 코스 · 495 POI · 43 가격 품목 · 4개 언어 전 경로 구현 · PWA 설치 지원
- **데이터 활용 적절성 (20점)** — 공공데이터포털 **두 건 실증** (TourAPI + 부천시 개인서비스요금) × **5개 엔드포인트 × 4개 언어 = 20 조합**
- **서비스 발전성 (20점)** — 관리자 CMS MVP 동작 중 + 광역 15개 시도 확장 로드맵 문서화

### 지역 특화 가점 (+2점)

- 부천시(서울 제외) 1개 지역 특화로 가점 대상에 해당
- RTO 특별상은 경기도 제외이므로 포기, 대신 가점 확보에 집중

### 최종 심사 (PT) 100점

- 완성도 (30점) · 실용성 (25점) 영역에서 **라이브 서비스 운영 중 + 관리자 CMS 실제 사용 시연** 강조

## 8. 확장·유지보수 시 불변 원칙

**양보 불가 (공모전 근거 직결):**

1. POI 데이터는 반드시 **TourAPI 기반**. Kakao/Naver 등 벤더 POI API는 보조로도 쓰지 않는다 (공모전 채점 근거 상실 방지)
2. 가격 공공데이터가 존재하는 지역은 해당 지역 데이터셋을 **정식 소스**로 사용한다. 내부 카탈로그로 대체하지 않는다
3. 지도 렌더링 SDK(Kakao Maps JS)는 **시각화 레이어만** 담당. POI·주소 검색 등 데이터성 기능에는 사용하지 않는다

**운영 편의:**

- 광역 시도 확장 시: 해당 시도 `personal-service-fee` OpenAPI 존재 확인 → 있으면 sync 스크립트 복제 작성, 없으면 해당 지역은 확장 대상에서 제외
- 관리자 커스텀 클레임 배포 절차는 `scripts/set-admin-claim.mjs` 사용 (서비스 계정 JSON 필요)
